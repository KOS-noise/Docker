# 세션 1 — Docker 개념 및 VM vs Container


## 1-1. 왜 Docker인가?

### 전통적인 배포 문제점 (Pre-Docker Era)

- **"내 PC에서는 되는데요"**: 개발자의 PC 환경(OS 버전, 라이브러리 설정 등)이 운영 서버와 달라서 발생하는 잦은 오류.
- **환경 불일치**: 개발 -> 테스트 -> 운영 서버로 갈수록 환경이 조금씩 달라져 배포할 때마다 '기도'해야 하는 상황.
- **자원 충돌**: 서버 한 대에 여러 앱을 띄울 때, 앱 A(Java 11)와 앱 B(Java 17)가 서로 다른 라이브러리 버전을 요구하여 충돌 발생.

### 실무에서 느끼는 Docker의 위력

- **온보딩(Onboarding)의 기적**: 
    - **과거**: 신입 개발자가 출근하면 개발 환경 설정에만 **이틀(2일)**이 걸림. 
    - **현재**: Docker만 설치되어 있다면 명령어 한 줄로 **5분** 만에 모든 개발 환경 세팅 완료.
- **환경의 표준화**: 내 노트북에서 잘 돌아갔다면, 전 세계 어디에 있는 서버에서도 **100% 동일하게** 동작함.

### 💡 세상을 바꾼 "컨테이너 박스"의 비유

도커(Docker)의 이름과 아이콘(고래 위 컨테이너)에는 깊은 뜻이 담겨 있음.

- **과거의 물류**: 배에 물건을 실을 때 피아노, 과일, 옷을 제각각 쌓아야 해서 선적 속도가 느리고 파손이 잦았음.
- **물류 혁명**: 규격화된 **'컨테이너 박스'** 등장 $\rightarrow$ 어떤 물건이든 박스에 넣기만 하면 배, 기차, 트럭 어디든 규격에 맞춰 착착 실을 수 있게 됨.
- **Docker의 혁명**: 자바, 파이썬, DB 등 어떤 소프트웨어든 **'도커 컨테이너'**라는 규격에 넣기만 하면, 어떤 서버든 상관없이 똑같은 방식으로 실행 가능함.

### Docker가 해결하는 것

- **Lego 블록식 조립**: 애플리케이션 + 가동 환경을 **하나의 단위(이미지)**로 묶어 관리함.
- **어디서든 동일 실행**: 환경이 통째로 복제되므로 서버 간 이동이 자유로움.
- **Java 버전 걱정 끝 (예시)**: 
    - 내 앱은 **Java 11**, 서버는 **Java 17**인 상황? 
    - 기존에는 충돌이 났지만, 도커는 이미지 안에 Java 11을 통째로 넣어 가기 때문에 서버 환경에 상관없이 **항상 Java 11**로 동작함.
- **가볍고 빠른 격리**: 컴퓨터 한 대를 통째로 빌리는(VM) 것보다 훨씬 가볍고 빠르게 필요한 만큼만 자원을 격리함.

## 1-2. VM vs Container 아키텍처 비교

### 먼저 "자원을 빌려쓴다"는 개념부터

컴퓨터에는 **CPU, 메모리, 디스크**라는 자원이 존재.
프로그램을 실행 = 이 자원을 OS로부터 빌려 씀.

문제는 **앱마다 원하는 환경이 다름.**

> 앱 A는 Java 11 필요, 앱 B는 Java 17 필요
> 앱 A는 Linux 환경에서만 동작

이 문제를 해결하는 방식이 **VM** 과 **Container** 두 가지.

---

### VM — "컴퓨터 안에 컴퓨터를 통째로 만든다"

내 컴퓨터(Windows) 위에 **가짜 컴퓨터(Linux)**를 통째로 생성하는 방식.

```
┌────────────────────────────────────────┐
│  가짜 컴퓨터 (Guest)                    │
│  ┌──────────────────────────────────┐  │
│  │  내 앱 (App)                     │  │
│  ├──────────────────────────────────┤  │
│  │  운영체제 전체 (Linux OS)         │  │  ← 수 GB 차지
│  └──────────────────────────────────┘  │
├────────────────────────────────────────┤
│  가짜 하드웨어를 만들어주는 소프트웨어   │
│  (Hypervisor: VMware, VirtualBox...)   │
├────────────────────────────────────────┤
│  내 실제 컴퓨터 (Windows + 하드웨어)    │
└────────────────────────────────────────┘
```

**비유:** 집 안에 조립식 가건물 신축 — 바닥·전기·수도 모두 독립.
완전히 분리되지만 **자원 소모가 큼.**

- VM 시작 = 컴퓨터 한 대 부팅 → **수십 초 ~ 수 분**
- 이미지 크기 = OS 전체 포함 → **수 GB**

---

### WSL도 VM과 같은 원리

**WSL(Windows Subsystem for Linux)** = Windows 위에서 Linux 명령어를 쓸 수 있게 해주는 도구.
내부적으로 경량 Linux VM을 띄워서 자원을 빌려주는 방식.

```
Windows 위에서 Linux 명령어가 동작하는 이유
= Windows가 내부에서 Linux VM을 조용히 실행해주기 때문
```

Docker Desktop(Windows)도 WSL2 위에서 동작.

```
실행 순서: Windows → WSL2(Linux VM) → Docker
```

WSL2가 없으면 Docker가 Linux 자원을 빌릴 수 없어서 실행 불가.

---

### Container — "앱 실행에 필요한 것만 쏙 빼서 격리한다"

OS를 통째로 만들지 않음.
**이미 있는 OS의 자원을 빌려 쓰되, 서로 안 보이게 커튼을 치는** 방식.

```
┌──────────────┬──────────────┬────────────────┐
│  Container A │  Container B │  Container C   │
│  앱 + 라이브러리│ 앱 + 라이브러리│ 앱 + 라이브러리 │  ← OS 없음, MB 단위
│  (커튼 안)   │  (커튼 안)   │   (커튼 안)    │
├──────────────┴──────────────┴────────────────┤
│           Docker Engine                      │  ← 커튼 관리자
├──────────────────────────────────────────────┤
│           OS 커널 (공용)                      │  ← 자원 공유
│           실제 하드웨어                        │
└──────────────────────────────────────────────┘
```

**비유:** 큰 사무실에 팀마다 **파티션(칸막이)** 설치.
전기·냉난방·화장실은 공유, 각 팀의 업무 내용은 서로 안 보임.
방을 새로 짓는 게 아니라 칸막이만 치므로 **빠르고 가벼움.**

- Container 시작 = 프로세스 하나 실행 → **1~2초**
- 이미지 크기 = 앱 + 필요한 라이브러리만 → **수십 ~ 수백 MB**

---

### 핵심 차이 한눈에 보기

| 항목 | VM | Container |
|------|----|-----------|
| 비유 | 집 안에 가건물 신축 | 사무실에 파티션 설치 |
| OS 포함 여부 | OS 전체 포함 | OS 없음 (공유) |
| 시작 시간 | 수십 초 ~ 수 분 | 1~2초 |
| 크기 | 수 GB | 수십 ~ 수백 MB |
| 격리 수준 | 완전 독립 | 커널은 공유 |
| Windows에서 Docker | WSL2(Linux VM) 위에서 동작 | — |

> **핵심 메시지:** VM은 "집 안에 집"을 짓는 것, Container는 "칸막이"로 공간을 나누는 것

## 1-3. Linux Kernel 격리 기술

Docker는 Linux 커널의 두 가지 기능을 활용함.

### Namespaces — "무엇을 볼 수 있는가" 격리

| Namespace | 격리 대상 |
|-----------|----------|
| PID | 프로세스 ID |
| NET | 네트워크 인터페이스 |
| MNT | 파일시스템 마운트 |
| UTS | 호스트명 |
| IPC | 프로세스 간 통신 |
| USER | 사용자/그룹 ID |

### cgroups — "얼마나 쓸 수 있는가" 제한

- CPU, 메모리, 디스크 I/O, 네트워크 대역폭 제한
- 예: 컨테이너 A는 CPU 최대 50%, 메모리 512MB

## 1-4. Docker Engine 아키텍처

```
docker CLI (클라이언트)
    │  REST API (Unix socket / TCP)
    ▼
Docker Daemon (dockerd)
    │
    ▼
containerd  ← 컨테이너 라이프사이클 관리
    │
    ▼
runc  ← 실제 컨테이너 생성 (OCI 표준)
```

- **docker CLI**: 사용자가 입력하는 명령어 도구
- **dockerd**: 백그라운드 데몬, 이미지·컨테이너 관리
- **containerd**: 컨테이너 런타임 표준 구현체
- **runc**: 커널 호출로 실제 격리 실행

## 1-5. 실습 — 따라하기

### Step 1. Docker 환경 확인

```bash
# Docker 버전 확인
docker version
```

확인 사항:
- `Client` 와 `Server` 두 섹션이 모두 나오면 정상
- Server가 안 나오면 Docker Desktop이 실행 안 된 상태 → 트레이에서 Docker 아이콘 확인

```bash
# Docker 시스템 전체 정보
docker info
```

확인 사항:
- `Server Version` — 설치된 Docker 버전
- `Total Memory` — Docker가 사용할 수 있는 메모리 (WSL2 할당량)
- `Operating System` — Docker 내부 OS (Linux 계열로 표시됨을 확인)

---

### Step 2. 첫 번째 컨테이너 실행

```bash
docker run hello-world
```

정상 출력 예시:
```
Hello from Docker!
This message shows that your installation appears to be working correctly.
...
```

**실행 흐름 따라가기:**
```
1. docker run hello-world 입력
2. 로컬에 hello-world 이미지 없음 → Docker Hub에서 자동 pull
3. 이미지로 컨테이너 생성
4. 컨테이너 안에서 메시지 출력 프로세스 실행
5. 프로세스 종료 → 컨테이너 자동 stopped
```

---

### 💡 도커 필수 기초 명령어 (Basics)

컨테이너의 생명주기를 관리하고 실습 문제를 풀 때 꼭 알아야 하는 핵심 명령어 모음.

| 구분 | 명령어 | 주요 옵션 및 설명 |
|:---:|:---|:---|
| **실행 및 생성** | `docker run [이미지]` | **컨테이너를 생성하고 즉시 시작** <br> `-d`: 백그라운드에서 실행 (터미널을 계속 사용 가능) <br> `--name [이름]`: 컨테이너에 원하는 이름을 부여 <br> `-p [PC포트]:[컨테이너포트]`: 호스트와 컨테이너의 포트 연결 <br> `--memory=[용량]`: 메모리 사용량 제한 (예: `64m`, `1g`) |
| | `docker stop [이름]` | 실행 중인 컨테이너를 안전하게 정지 |
| | `docker rm [이름]` | 컨테이너 삭제 (`-f` 추가 시 실행 중이어도 강제 삭제) |
| **조회 및 확인** | `docker ps` | 실행 중인 컨테이너 목록 (`-a`: 종료된 항목까지 모두 표시) |
| | `docker images` | 로컬 환경에 다운로드된 이미지 목록 확인 |
| | `docker logs -f [이름]` | 컨테이너 내부의 로그를 실시간으로 확인 |
| **자원 및 이미지** | `docker stats` | 실행 중인 컨테이너들의 자원(CPU, 메모리) 사용량 실시간 모니터링 |
| | `docker rmi [이미지]` | 로컬 이미지 삭제 (태그가 있다면 `이미지:태그` 형식 사용) |

---

### Step 3. 컨테이너가 남아있는지 확인

```bash
# 실행 중인 컨테이너 목록
docker ps

# 종료된 컨테이너 포함 전체 목록
docker ps -a
```

`hello-world` 컨테이너가 `Exited` 상태로 남아있는 것 확인.
→ 컨테이너는 종료돼도 기록이 남음. `docker rm`으로 삭제 가능.

```bash
# 컨테이너 ID 확인 후 삭제
docker rm $(docker ps -aq)

# 다시 확인 — 목록이 비어있어야 함
docker ps -a
```

---

### Step 4. 이미지 목록 확인

```bash
docker images
```

`hello-world` 이미지가 로컬에 저장된 것 확인.
→ 이미지는 컨테이너를 삭제해도 남아있음. 다시 run하면 pull 없이 바로 실행.

```bash
# 이미지 삭제
docker rmi hello-world

# 다시 확인
docker images
```

---

### Step 5. 자원 사용량 실시간 확인

컨테이너를 하나 띄워놓고 자원 사용량 확인.

```bash
# nginx 컨테이너 백그라운드 실행
docker run -d --name test-nginx nginx:alpine

# 실시간 자원 사용량 (CPU, 메모리)
docker stats

# 나가기: Ctrl + C
```

출력에서 확인할 것:
- `MEM USAGE` — 컨테이너가 실제 사용 중인 메모리
- `CPU %` — CPU 점유율 (거의 0%에 가까움 → 가볍다는 것 체감)

```bash
# 정리
docker stop test-nginx
docker rm test-nginx
```

### 문제 1 — 같은 이미지로 컨테이너 2개 동시 실행

`nginx:alpine` 이미지로 컨테이너 **2개 동시 실행**

조건:
- 첫 번째: 이름 `web-1`, 포트 `8081:80`
- 두 번째: 이름 `web-2`, 포트 `8082:80`
- 브라우저에서 `http://localhost:8081` 과 `http://localhost:8082` 둘 다 접속 확인
- `docker ps` 로 두 컨테이너가 동시에 실행 중인 것 확인
- 확인 후 두 컨테이너 모두 정지 및 삭제

> 힌트: 이미지 하나로 컨테이너를 몇 개든 만들 수 있음. 포트만 겹치지 않으면 됨.

---

### 문제 2 — 메모리 제한을 걸고 자원 사용량 비교

컨테이너에 메모리 제한을 걸고, `docker stats`로 실제 적용 여부 확인

조건:
- 이미지: `nginx:alpine`
- 컨테이너 이름: `limited`
- 메모리 제한: `64m` (64MB)
- `docker stats limited` 로 `MEM LIMIT` 이 `64MiB` 로 표시되는지 확인

> 힌트: `docker run` 의 `--memory` 옵션 사용

---
